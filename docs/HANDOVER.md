# 두산 경기 반자동 포스팅 시스템 인수인계 문서

## 1. 프로젝트 목적

이 시스템은 두산베어스 경기 종료 후 네이버 블로그용 포스팅 초안을 자동 생성하고, 사용자가 직접 검토 및 복붙 발행할 수 있도록 돕는 반자동 운영 도구입니다.

핵심 목표는 아래와 같습니다.

- 경기 결과 및 주요 정보 수집
- 사용자 문체를 반영한 블로그 초안 생성
- 이미지 후보 정리
- 복붙하기 쉬운 웹 UI 제공

자동 발행이 아닌 반자동 방식을 채택한 이유는 네이버 블로그의 범용 포스팅 API 제약과 계정 리스크 때문입니다.

## 2. 현재 권장 아키텍처

초기 제안은 `Python + FastAPI + SQLite`였지만, Supabase 배포를 전제로 하면 아래 구조가 더 적합합니다.

- DB: Supabase Postgres
- 스케줄러: Supabase Cron
- 서버 로직: Supabase Edge Functions
- 이미지 후보 메타: Postgres + Storage
- UI: 별도 웹 프론트 또는 로컬 정적 페이지

### 이유

- Supabase는 Python 웹 서버 호스팅보다는 Postgres/Storage/Edge Functions 조합에 최적화되어 있습니다.
- Edge Functions는 TypeScript/Deno 기반입니다.
- 주기 실행은 `pg_cron`과 HTTP 호출이 가장 단순합니다.
- 로컬 검토형 UX는 프론트만 별도로 두는 편이 운영상 안전합니다.

## 3. MVP 범위

이번 버전의 MVP 범위는 아래까지입니다.

- 두산 경기 일정 확인
- 종료 경기 결과 저장
- 포스트 제목/요약/본문/태그 생성
- 이미지 후보 링크/출처 정리
- 웹 UI에서 복사 버튼 제공
- 발행 상태 관리

아래는 제외합니다.

- 네이버 블로그 자동 로그인
- 네이버 블로그 자동 발행
- 이미지 자동 업로드
- 주간 통계 대시보드

## 4. 주요 데이터 흐름

1. Cron이 경기 종료 시간대에 `process-completed-games` 함수를 호출합니다.
2. `process-completed-games`는 해당 날짜의 두산 경기 상태를 확인합니다.
3. 종료 경기라면 `games` 테이블에 경기 메타를 저장합니다.
4. 포스트가 아직 없으면 `generate-post` 함수로 초안을 생성합니다.
5. 이미지 후보가 없으면 `fetch-image-candidates` 함수로 링크 후보를 저장합니다.
6. 사용자는 웹 UI에서 제목/본문/태그를 검토하고 복사합니다.
7. 최종 발행 후 `published` 상태로 바꿉니다.

## 5. 사용자 문체 반영 규칙

문체는 아래 원칙을 지켜야 합니다.

- 존댓말
- 너무 딱딱하지 않은 팬 블로그 톤
- 인사말로 자연스럽게 시작
- `~인데요`, `~같습니다`, `~보입니다`를 자연스럽게 사용
- `ㅎㅎ`, `ㅎㅎㅎ`를 과하지 않게 삽입
- 기사체 문장 금지
- 단정형보다는 의견형 표현 선호

예시 도입부:

`안녕하세요 토끼돼지입니다~~~!`

## 6. 데이터 모델

### `games`

경기 원본/정규화 데이터 저장용입니다.

- 경기 날짜
- 시즌
- 상대팀
- 홈/원정
- 상태
- 최종 점수
- 승패
- 선발투수
- 주요 선수
- 주요 장면
- 원본 JSON

### `posts`

생성 결과와 사용자 편집본을 함께 저장합니다.

- 제목
- 한줄 요약
- 본문
- 태그
- 상태
- 생성 시각
- 수정 시각

### `image_candidates`

자동 수집한 이미지 후보의 메타 정보만 저장합니다.

- 원본 URL
- 썸네일 URL
- 출처명
- 출처 URL
- 복사용 출처 메모

### `settings`

운영자 스타일 및 스케줄 설정 저장용입니다.

- 팀명
- 문체 프롬프트
- 이미지 소스 정책
- 스케줄 타임존
- 생성 활성화 여부

## 7. Edge Functions 역할

### `fetch-kbo-games`

역할:

- 특정 날짜 또는 최근 일정 기준으로 두산 경기 조회
- 종료/취소/예정 상태 판별
- 경기 메타 정규화
- `games` upsert

주의:

- 더블헤더 고려 필요
- 우천 취소 상태 저장 필요
- 같은 경기 중복 저장 방지 필요

### `generate-post`

역할:

- 경기 JSON과 스타일 프롬프트를 기반으로 포스트 생성
- 제목/요약/본문/태그 반환
- `posts` upsert

주의:

- 데이터 부족 시 생성 보류
- 루머성 문장 금지
- 매 경기 완전 동일한 템플릿 방지

### `fetch-image-candidates`

역할:

- 공식 채널 또는 허용 범위가 명확한 링크를 수집
- 이미지 URL과 출처 메타 저장

주의:

- 무단 재업로드 기능 구현 금지
- 썸네일/링크/출처 표시 위주

### `process-completed-games`

역할:

- 전체 워크플로 오케스트레이션
- 종료 경기만 골라 후속 함수 호출
- 중복 생성 방지

## 11. 실제 운영 시 주의점

- 경기 데이터 소스가 변경되면 정규화 로직부터 깨질 수 있습니다.
- 경기 종료 판정이 애매한 경우 연장전/우천 중단 이슈를 먼저 확인해야 합니다.
- 생성 문구는 블로그 운영자가 직접 손보는 전제를 유지해야 합니다.
- 이미지 사용 범위는 반드시 출처 정책을 따르도록 유지해야 합니다.

## 13. 결론

이 프로젝트는 Supabase에 "그대로 Python 서버를 올리는 방식"보다는, `Supabase를 백엔드로 사용하고 UI를 분리하는 방식`으로 구현하는 것이 맞습니다.

즉, 업로드는 가능합니다. 다만 정확히는 아래처럼 이해하는 것이 맞습니다.

- Supabase에 올릴 것: DB, 스케줄, 서버 함수, 저장소
- 별도로 둘 것: 사용자가 보는 웹 UI

이 구조가 현재 요구사항과 운영 리스크를 가장 잘 맞춥니다.
